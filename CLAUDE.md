# 반도체 시료 생산주문관리 시스템 (SampleOrderSystem)

이 폴더는 개인과제의 **최종 구현 폴더**다. 반도체 시료 생산 주문을 등록·승인·생산·출고까지 관리하는 콘솔 애플리케이션.

## 문서 구조 — 어디를 봐야 하는가

CLAUDE.md는 요구사항이나 도메인 규칙을 반복 설명하지 않는다. **"무엇을 만드는가"는 아래 문서를 참고하고, 이 문서는 "어떻게 작업하는가"(오케스트레이션/개발 방법론)만 다룬다.**

- **기능 요구사항, 도메인 규칙, 사용자 시나리오, 완료 기준** → [`PRD.md`](./PRD.md)
- **구현 진행 이력, 각 TDD 사이클의 범위/완료 조건** → [`plan/전체플랜.md`](./plan/전체플랜.md), `plan/plan_N.md`
- **원본 과제 요구사항 원문**(PDF, 페이지 이미지) → `docs/` 폴더
- **실행 방법, 프로젝트 구조, 테스트 실행법** → [`README.md`](./README.md)

## 구현 참고 사항

- `ConsoleMVC` 폴더는 Model / Controller / View 패키지 구조와 역할 분리를 보여주는 POC이므로, 본 시스템도 콘솔 기반 + MVC 구조 분리를 참고해서 구현한다.
- 화면(UI) 레이아웃은 PDF의 예시 화면을 참고하되 자유롭게 구성 가능 (필수 요구사항 아님).

## 개발 방법론 — Agentic Engineering

본 구현은 기능 명세를 충족하는 것을 넘어 **고품질 코드**를 목표로 한다. 아래 5가지를 항상 주안점으로 삼는다. Claude(에이전트)는 매 작업에서 이 5가지를 반드시 유념한다.

### 1. 문서 관리 (`CLAUDE.md`, `PRD.md` 등)

- `CLAUDE.md`(본 문서): 개발 방법론의 단일 진실 공급원(SSOT). 요구사항 자체는 다루지 않는다 — 방법론이 바뀌면 가장 먼저 갱신한다.
- `PRD.md`: 기능 범위, 도메인 규칙, 사용자 시나리오, 완료 기준을 정리한 제품 요구사항 문서.
- `plan/` 폴더: 작업 계획 관리
  - `plan/전체플랜.md` — 전체 로드맵과 마일스톤
  - `plan/plan_N.md` — 사이클별 세부 Plan (아래 TDD Red-Green-Review 사이클의 "Plan" 단계 산출물). 매 사이클 시작 전 사람 파트너에게 받고 검토 후 진행한다.

### 2. Harness (에이전트 하네스) 도입

**정의**: AI Agent에서 "하네스(Harness)"란, Agent가 **장시간(long-horizon)** 실행되더라도 올바른 임무를 계속 수행하도록 만드는 방법론이다. 에이전트 자체의 성능(모델 능력)을 높이는 것이 아니라, **Workflow 레벨에서 안전장치를 설계**하는 것이 하네스 엔지니어링이다. 핵심 키워드는 **장시간 실행** — 에이전트가 오래, 여러 사이클에 걸쳐 작업해도 목표에서 이탈하지 않도록 막는 것이다.

**본 프로젝트에서의 적용:**

- **작게 쪼갠 Plan 단위** (`plan/plan_N.md`)로 작업을 잘게 나눠, 에이전트가 한 번에 너무 큰 범위를 폭주하며 구현하지 않도록 한다. 각 사이클은 하나의 좁은 목표만 가진다.
- **Human-in-the-Loop 게이트**: RED 확인, GREEN 확인, 그리고 무엇보다 **사람 파트너의 REVIEW 승인 없이는 GREEN을 커밋하지 않는다**는 규칙 자체가 에이전트 폭주를 막는 하네스다. (아래 3번 Test 항목 참고)
- **커밋 단위의 체크포인트**: RED/GREEN을 분리 커밋해, 에이전트가 잘못된 방향으로 갔을 때 언제든 마지막 정상 체크포인트로 되돌릴 수 있게 한다.
- **문서(CLAUDE.md/PRD.md/Plan.md)를 통한 목표 고정**: 매 사이클 시작 시 Plan을 먼저 검토받아, 에이전트가 스스로 목표를 재해석/확장하지 않고 합의된 범위 안에서만 움직이도록 고정한다.
- 즉, 이 프로젝트의 하네스는 "pytest를 돌리는 인프라"가 아니라, **Plan 단위 쪼개기 + Human 승인 게이트 + 체크포인트 커밋**으로 구성된 워크플로우 안전장치를 의미한다.

### 3. Test — TDD Red-Green-Review 사이클

- `.claude/skills/tdd/SKILL.md`의 **Red-Green-Review** 워크플로우를 예외 없이 따른다.
- 순서: Plan(`plan/plan_N.md`) 작성/검토 → **RED**(실패하는 테스트 작성 및 확인) → **RED 커밋** → **GREEN**(최소 구현, 테스트 통과 확인만, 커밋 보류) → **사람 파트너 REVIEW** → 승인 시 **GREEN 커밋** → 다음 목표로 반복.
- 실패하는 테스트 없이 프로덕션 코드를 작성하지 않는다. 사람 리뷰 없이 GREEN을 커밋하지 않는다.
- 단, 사람 파트너가 "fast-fix"로 명시한 작업은 RED/GREEN을 분리하지 않고 테스트+구현을 함께 작성해 바로 커밋할 수 있다 (커밋 메시지에 `fast-fix:` 접두사 사용).
- 콘솔 View/Controller의 실제 `input()`/`print()`는 외부 경계로 보고 pytest 대상에서 제외, 매 사이클 수동 스모크 테스트로 확인한다.

### 4. Clean Code

- YAGNI: Plan 범위를 벗어난 기능/추상화를 미리 만들지 않는다.
- 역할 분리(MVC 등 구조 분리) 유지, 중복보다 과도한 추상화를 더 경계한다.
- 리뷰 단계에서 스코프 크리프와 과도한 설계 여부를 항상 점검한다.
- 린트/포맷: `ruff`가 로컬 환경에 설치되어 있음. 프로젝트 레벨 설정 파일(`pyproject.toml`/`ruff.toml`)은 두지 않고, 에디터의 전역 format-on-save 설정에 맡긴다 (별도 프로젝트 설정 관리 안 함).
- 서브에이전트(`.claude/agents/doc-checker.md`, `bug-hunter.md`)를 활용해 문서-코드 정합성과 엣지 케이스를 주기적으로 점검하고, `sub-agents-reports/`에 결과를 기록한 뒤 결함은 별도 사이클로 수정한다.

### 5. Commit 이력

- RED 커밋과 GREEN 커밋을 분리한다 (한 커밋에 섞지 않는다). fast-fix는 예외.
- 커밋 메시지에 사이클 목표를 명확히 남긴다: `RED: <목표> 실패하는 테스트 추가`, `GREEN: <목표> 구현 (리뷰 완료)`, `fast-fix: <목표>`.
- 커밋 이력 자체가 개발 과정(계획→실패→구현→리뷰→통과)을 설명하는 문서가 되도록 관리한다.
