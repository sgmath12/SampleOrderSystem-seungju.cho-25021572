# 반도체 시료 생산주문관리 시스템 (SampleOrderSystem)

이 폴더는 개인과제의 **최종 구현 폴더**다.

## 배경

가상의 반도체 회사 "S-Semi"는 다양한 반도체 시료(Sample)를 생산해 연구소, 팹리스(Fabless) 업체, 대학 연구실 등에 납품한다. 시료는 주문이 들어오면 웨이퍼 공정 설비를 통해 제작되고 검수를 거쳐 고객에게 출고된다.

기존에는 엑셀/메모장으로 주문을 관리해 실수가 잦고 재고·공정 현황을 한눈에 파악하기 어려웠다. 이를 해결하기 위해 "반도체 시료 생산주문관리 시스템"을 개발한다.

## 시스템 개요

- **콘솔 기반**으로 동작 (담당자가 직접 명령 입력)
- 핵심 흐름: 시료를 등록하고 → 주문을 처리
- **생산라인**
  - 공장에서 시료 하나를 생산하는 설비 흐름 (단일 라인)
  - 하나의 생산라인은 시료를 하나씩 생산
  - 주문이 들어온 시료에 대해서만 생산

## 역할별 흐름 (`6page.png` 참고)

고객(시료 요청자) → [시료 요청/이메일] → 주문 담당자(주문서 관리) → [주문서 전달] → 생산 담당자(시료 생산·승인) → [승인/거절] → 주문 담당자

## 주문 상태 흐름 (`7page.png` 참고)

모든 주문은 아래 상태 중 하나를 가진다.

| 상태 | 의미 |
|---|---|
| RESERVED | 주문 접수 |
| REJECTED | 주문 거절 (정상 흐름 밖의 상태, **모니터링에서 제외**) |
| PRODUCING | 주문 승인 완료 + 재고 부족으로 생산 중 |
| CONFIRMED | 주문 승인 완료 + 출고 대기 중 |
| RELEASE | 출고 완료 |

상태 전이 다이어그램:

```
주문 등록 → RESERVED
RESERVED --거절--> REJECTED
RESERVED --승인, 재고 확인-->
    재고 충분 → CONFIRMED → (출고 처리) → RELEASE
    재고 부족 → PRODUCING → (생산 완료) → CONFIRMED → (출고 처리) → RELEASE
```

## 메인 메뉴 (`11page.png` 예시 UI)

기능별 선택 화면을 표시하고 전체 시료에 대한 요약 정보를 보여준다. 화면 구성은 자유롭게 결정.

| 메뉴 | 의미 |
|---|---|
| 시료 관리 | 새로운 시료 등록, 목록 조회, 이름 검색 |
| 주문 (접수/승인/거절) | 고객 주문 접수 및 생산라인 담당자의 승인·거절 처리 |
| 모니터링 | 상태별 주문 수 및 시료별 재고 현황 확인 |
| 출고 처리 | CONFIRMED 상태 주문에 대해 출고 실행 |
| 생산라인 | 현재 생산 중인 시료 및 대기 중인 생산 큐 확인 |

## 기능 명세

### 1) 시료 관리 (`13page.png`)

- 시료(Sample)는 시스템의 가장 기본이 되는 단위. 각 시료는 고유한 이름과 속성을 가지며, **시스템에 등록된 시료만 주문 가능**.
- **수율** = 정상적인 시료 / 총 생산 시료 (ex. 100개 생산 중 정상품 90개 = 0.9)
- 시료 등록: 새 시료를 시스템에 추가. 속성값 — 시료 ID, 이름, 평균 생산 시간, 수율
- 시료 조회: 등록된 모든 시료 목록 확인, 현재 재고 수량도 함께 표시
- 시료 검색: 이름 등 속성으로 특정 시료 검색

### 2) 시료 주문 (`15page.png`)

- 고객이 시료를 요청하면 주문 담당자가 주문을 생성 (시료 예약)
- 고객이 원하는 시료와 수량을 주문 → 이 시점의 주문 상태는 **RESERVED**
- 예약 시 입력값: 시료 ID, 고객명, 주문 수량

### 3) 주문 승인/거절 (`17page.png`)

- 접수된 주문(RESERVED) 목록 확인, 특정 주문에 대해 승인 또는 거절
- **주문 승인**: 재고 상황에 따라 자동으로 2가지 방식으로 처리
  - 재고 충분 → 주문을 즉시 **CONFIRMED** 상태로 전환
  - 재고 부족 → 생산라인에 자동 등록, 주문 상태를 **PRODUCING**으로 전환
- **주문 거절**: 즉시 **REJECTED** 상태로 전환

### 4) 모니터링 (`19page.png`)

- **주문량 확인**: 현재 상태별(RESERVED/CONFIRMED/PRODUCING/RELEASE) 목록 확인. REJECTED는 유효한 주문이 아니므로 무시.
- **재고량 확인**: 각 시료별 현재 재고 수량 확인. 주문 대비 재고 수량에 따라 상태 표기
  - 여유: 주문 대비 재고 충분
  - 부족: 주문 대비 재고 수량 부족
  - 고갈: 재고 수량이 0

### 5) 생산라인 (`21page.png`)

- 생산라인에 대한 정보 표시
- 주문량에 대한 부족분을 생산하되, 수율 및 오차를 고려하여 시료 생산
  - **실생산량** = ceil(부족분 / 수율)
  - **총 생산 시간** = 평균 생산 시간 × 실생산량
- 생산 완료 시 주문 상태 PRODUCING → CONFIRMED 로 변경
- 생산 현황 표기: 현재 생산 중인 시료에 대한 정보 (표기 수준은 자율 결정, ex. 주문 정보, 현재까지의 생산량 등)
- 대기 주문 확인: 생산라인의 대기열(생산 큐) 이용, 대기 목록 출력
  - **스케줄링 전략: FIFO** (선입선출)

### 6) 출고 처리 (`23page.png`)

- 재고가 충분해진 CONFIRMED 주문에 대해 출고 처리
- 특정 주문에 대해 출고 실행 → 주문 상태가 **RELEASE**로 전환

## 구현 참고 사항

- `ConsoleMVC` 폴더는 Model / Controller / View 패키지 구조와 역할 분리를 보여주는 POC이므로, 본 시스템도 콘솔 기반 + MVC 구조 분리를 참고해서 구현한다.
- 화면(UI) 레이아웃은 PDF의 예시 화면을 참고하되 자유롭게 구성 가능 (필수 요구사항 아님).

## 개발 방법론 — Agentic Engineering

본 구현은 기능 명세를 충족하는 것을 넘어 **고품질 코드**를 목표로 한다. 아래 5가지를 항상 주안점으로 삼는다. Claude(에이전트)는 매 작업에서 이 5가지를 반드시 유념한다.

### 1. 문서 관리 (`CLAUDE.md`, `PRD.md` 등)

- `CLAUDE.md`(본 문서): 요구사항 스펙과 개발 방법론의 단일 진실 공급원(SSOT). 요구사항/방법론이 바뀌면 가장 먼저 갱신한다.
- `PRD.md`: 제품 요구사항 문서. 기능 범위, 사용자 시나리오, 완료 기준을 정리한다. (아직 없다면 구현 시작 전 작성)
- `plan/` 폴더: 작업 계획 관리
  - `plan/전체플랜.md` — 전체 로드맵과 마일스톤
  - `plan/plan_N.md` — 사이클별 세부 Plan (아래 TDD Red-Green-Review 사이클의 "Plan" 단계 산출물). 매 사이클 시작 전 사람 파트너에게 받고 검토 후 진행한다.

### 2. Harness (에이전트 하네스) 도입

**정의**: AI Agent에서 "하네스(Harness)"란, Agent가 **장시간(long-horizon)** 실행되더라도 올바른 임무를 계속 수행하도록 만드는 방법론이다. 에이전트 자체의 성능(모델 능력)을 높이는 것이 아니라, **Workflow 레벨에서 안전장치를 설계**하는 것이 하네스 엔지니어링이다. 핵심 키워드는 **장시간 실행** — 에이전트가 오래, 여러 사이클에 걸쳐 작업해도 목표에서 이탈하지 않도록 막는 것이다.

**본 프로젝트에서의 적용:**

- **작게 쪼갠 Plan 단위** (`plan/plan_N.md`)로 작업을 잘게 나눠, 에이전트가 한 번에 너무 큰 범위를 폭주하며 구현하지 않도록 한다. 각 사이클은 하나의 좁은 목표만 가진다.
- **Human-in-the-Loop 게이트**: RED 확인, GREEN 확인, 그리고 무엇보다 **사람 파트너의 REVIEW 승인 없이는 GREEN을 커밋하지 않는다**는 규칙 자체가 에이전트 폭주를 막는 하네스다. (아래 3번 Test 항목 참고)
- **커밋 단위의 체크포인트**: RED/GREEN을 분리 커밋해, 에이전트가 잘못된 방향으로 갔을 때 언제든 마지막 정상 체크포인트로 되돌릴 수 있게 한다.
- **문서(CLAUDE.md/PRD.md/Plan.md)를 통한 목표 고정**: 매 사이클 시작 시 Plan을 먼저 검토받아, 에이전트가 스스로 목표를 재해석/확장하지 않고 합의된 범위 안에서만 움직이도록 고정한다.
- 즉, 이 프로젝트의 하네스는 "pytest를 돌리는 인프라"가 아니라, **Plan 단위 쪼개기 + Human 승인 게이트 + 체크포인트 커밋**으로 구성된 워크플로우 안전장치를 의미한다.

### 3. Test — TDD Red-Green-Review 사이클

- `.claude/skills/tdd/SKILL.md`의 **Red-Green-Review** 워크플로우를 예외 없이 따른다.
- 순서: Plan(`plan/plan_N.md`) 작성/검토 → **RED**(실패하는 테스트 작성 및 확인) → **RED 커밋** → **GREEN**(최소 구현, 테스트 통과 확인만, 커밋 보류) → **사람 파트너 REVIEW** → 승인 시 **GREEN 커밋** → 다음 목표로 반복.
- 실패하는 테스트 없이 프로덕션 코드를 작성하지 않는다. 사람 리뷰 없이 GREEN을 커밋하지 않는다.

### 4. Clean Code

- YAGNI: Plan 범위를 벗어난 기능/추상화를 미리 만들지 않는다.
- 역할 분리(MVC 등 구조 분리) 유지, 중복보다 과도한 추상화를 더 경계한다.
- 리뷰 단계에서 스코프 크리프와 과도한 설계 여부를 항상 점검한다.
- 린트/포맷: `ruff`가 로컬 환경에 설치되어 있음. 프로젝트 레벨 설정 파일(`pyproject.toml`/`ruff.toml`)은 두지 않고, 에디터의 전역 format-on-save 설정에 맡긴다 (별도 프로젝트 설정 관리 안 함).

### 5. Commit 이력

- RED 커밋과 GREEN 커밋을 분리한다 (한 커밋에 섞지 않는다).
- 커밋 메시지에 사이클 목표를 명확히 남긴다: `RED: <목표> 실패하는 테스트 추가`, `GREEN: <목표> 구현 (리뷰 완료)`.
- 커밋 이력 자체가 개발 과정(계획→실패→구현→리뷰→통과)을 설명하는 문서가 되도록 관리한다.
