## bug-hunter 리포트

1. **[치명적] 시료 ID 중복 등록이 조용히 허용됨** — `SampleRepository.register()`가 `sample_id` 중복을 검사하지 않아, 같은 ID로 재등록하면 리스트에 중복 추가되고 `find_by_id()`는 항상 첫 번째 것만 참조. 정정 시도가 조용히 무시되고 모니터링에 유령 재고 행이 남음.
2. **[치명적] `avg_production_time` 값 검증 부재 → 극단적 음수 입력 시 크래시 가능** — 수율만 검증하고 평균생산시간은 검증하지 않아, 큰 음수 입력 시 `production_time`이 음수가 되고 `estimated_completion_at`이 UNIX epoch 이전 타임스탬프를 만들어 `datetime.fromtimestamp()`가 `OSError`를 던짐. `_safe_call`은 `ValueError`만 잡으므로 앱 전체가 크래시.
3. **[중간] 주문번호 일련번호가 날짜별로 리셋되지 않음** — `OrderRepository`의 `_next_id`가 전역 카운터라, 날짜가 바뀌어도 `0001`로 리셋되지 않고 계속 증가. PRD의 "당일 일련번호" 의도와 어긋남.
4. **[중간] `watch_current_progress` 중 Ctrl+C(KeyboardInterrupt) 미처리** — "아무 키나 누르면 중단"이라는 안내와 달리 Ctrl+C는 `msvcrt.kbhit()`으로 감지되지 않고 `time.sleep()` 중 즉시 `KeyboardInterrupt`를 발생시켜 `_safe_call`(ValueError만 캐치)을 뚫고 앱이 죽음.
5. **[경미] 페이지네이션 마지막 페이지에서 무효 N/P 입력 시 작업 전체가 취소됨** — 유효 범위 밖 네비게이션 키 입력이 "재입력 요청"이 아니라 "잘못된 번호" 처리로 이어져 승인/거절/출고 흐름이 처음부터 다시 시작해야 함.
6. **[경미/설계 모호] `inventory_status(0, 0)`이 항상 "고갈"로 표시** — 수요가 전혀 없는 갓 등록한 시료도 재고 0이면 무조건 "고갈"로 표시되어 관리자가 오인할 수 있음. PRD 문구 그대로 구현한 것이라 스펙 위반은 아님.
7. 확인 후 문제없음: 재고==주문수량 경계, 수율 0 나누기 방지(등록 시점 검증), 부족분/실생산량/잉여 계산 정확도, FIFO 큐 단일 인스턴스 공유, ValueError 기반 도메인 예외의 일관된 처리.
